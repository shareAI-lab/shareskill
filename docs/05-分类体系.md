# ShareSkill 分类体系

## 一、设计原则

### 1.1 用户友好

- 分类名称直观易懂
- 覆盖用户常见搜索场景
- 避免过于技术化的术语

### 1.2 分布均衡

- 避免单个分类过于集中（不超过 40%）
- 保持分类间有明确边界
- 允许边缘案例存在于 "other"

### 1.3 可扩展

- 新增分类不影响现有数据
- LLM 分类时有明确指引

## 二、分类定义

### 主分类（14 类）

| Key | 英文名 | 中文名 | 描述 | 典型场景 |
|-----|--------|--------|------|----------|
| `frontend` | Frontend | 前端开发 | Web/移动端 UI 开发 | React、Vue、CSS、组件库 |
| `backend` | Backend | 后端开发 | 服务端、API、数据库 | Node.js、Python、API设计 |
| `mobile` | Mobile | 移动开发 | iOS、Android、跨平台 | Swift、Kotlin、Flutter |
| `devtools` | Dev Tools | 开发工具 | 编辑器、CLI、Git 工作流 | Git、编辑器插件、脚手架 |
| `testing` | Testing | 测试 | 单元测试、E2E、QA | Jest、Playwright、测试策略 |
| `devops` | DevOps | 运维 | CI/CD、部署、监控 | Docker、K8s、GitHub Actions |
| `security` | Security | 安全 | 认证、加密、审计 | OAuth、加密、安全审查 |
| `ai` | AI & ML | AI 与机器学习 | LLM、模型训练、数据科学 | LLM调用、MCP、Agent开发 |
| `data` | Data | 数据处理 | 分析、ETL、可视化 | Pandas、SQL、图表生成 |
| `content` | Content | 内容创作 | 文档、翻译、写作 | Markdown、技术写作、翻译 |
| `design` | Design | 设计 | UI/UX、图形、创意 | Figma、图像生成、设计系统 |
| `productivity` | Productivity | 效率工具 | 自动化、工作流、任务管理 | 任务分解、自动化脚本 |
| `integration` | Integration | 集成 | API 连接器、第三方服务 | Slack、Notion、外部 API |
| `other` | Other | 其他 | 未分类 | 边缘案例 |

### 分类映射关系

从旧分类到新分类的迁移：
- `development` → 拆分到 `frontend`、`backend`、`devtools`、`mobile`
- 保留：`testing`、`devops`、`data`、`content`、`design`、`productivity`、`integration`、`other`
- 新增：`security`、`ai`

## 三、分类逻辑

### 3.1 LLM 分类 Prompt

```
分析这个 Agent Skill，选择最匹配的分类。

分类说明：
- frontend: Web/移动端 UI 开发，如 React、Vue、CSS、组件
- backend: 服务端开发，如 API、数据库、服务器
- mobile: iOS/Android/跨平台移动开发
- devtools: 开发工具，如 Git、CLI、编辑器
- testing: 测试相关，如单元测试、E2E、QA
- devops: 运维相关，如 CI/CD、Docker、部署
- security: 安全相关，如认证、加密、审计
- ai: AI/ML 相关，如 LLM、模型、Agent
- data: 数据处理，如分析、ETL、可视化
- content: 内容创作，如文档、翻译、写作
- design: 设计相关，如 UI/UX、图形、创意
- productivity: 效率工具，如自动化、工作流
- integration: 第三方集成，如 API 连接器
- other: 以上都不匹配

规则：
1. 选择最核心的一个分类
2. 如果涉及多个领域，选择主要功能所属类别
3. "代码审查" → devtools（不是 testing）
4. "API 开发" → backend（不是 integration）
5. "LLM 调用" → ai
6. 不确定时选 other
```

### 3.2 分类边界说明

| 场景 | 正确分类 | 说明 |
|-----|---------|-----|
| React 组件开发 | frontend | 主要是 UI 开发 |
| REST API 开发 | backend | 服务端逻辑 |
| API 连接器（调用第三方） | integration | 连接外部服务 |
| 测试框架使用 | testing | 测试相关 |
| 代码审查/Lint | devtools | 开发工具 |
| Git 工作流 | devtools | 开发工具 |
| Docker 部署 | devops | 运维 |
| LLM 提示词工程 | ai | AI 相关 |
| 数据分析脚本 | data | 数据处理 |
| Markdown 文档 | content | 内容创作 |
| UI 设计稿生成 | design | 设计 |
| 任务自动化 | productivity | 效率 |

## 四、数据迁移

### 4.1 迁移策略

1. 更新 LLM prompt 使用新分类
2. 对已有 skills 进行重新分类（批量 LLM 调用）
3. 更新前端分类展示

### 4.2 迁移 SQL

```sql
-- 添加新分类到 skill_categories 表
INSERT INTO skill_categories (key, label, hint, sort_order) VALUES

```

## 六、统计与监控

### 6.1 分类分布监控

- 目标：每个分类占比 5-30%
- 告警：单个分类超过 40%
- 定期检查 `other` 分类，识别潜在新分类

### 6.2 分类质量抽查

- 每周抽查各分类 10 个 skills
- 验证分类准确性
- 收集错误分类案例，改进 prompt
